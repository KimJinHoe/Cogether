## 6. 회고
### 기획 및 프로젝트 관리
```
기획 당시, 우리와 같은 주니어 개발자들에게 필요한 것이 무엇인지 깊이 고민하며 주니어 개발자 SNS라는 주제를 정했다.
기획부터 개발까지 사용자의 입장을 깊이 고려하여 기능과 UI/UX 디자인을 구상했다.
팀장 및 프론트를 맡아 프로젝트를 진행하게 되었다. Git과 Jira를 이용하고 관련 규칙, 브런치 전략을 사전에 정해 프로젝트 관리를 했다.
일일 스크럼을 진행하며 팀원들의 진행사항을 확인하고 매일 백로그의 진행 사항을 업데이트하여 바람직한 번다운차트를 만들기 위해 노력했다.
```
### Vue3에 대해
```
팀원의 희망으로 기존 학습한 Vue2가 아닌 Vue3를 사용하게 되었다. 같은 Vue지만 생각보다 많은 부분이 바뀌어서 학습하는데 시간이 오래 걸렸다.
Vue3의 가장 큰 장점인 Composition API은 코드의 가독성 향상, 재사용성의 개선, 타입 추론 등 기존보다 이점이 많았다.
대신, 이점만큼 많은 단점들이 존재했다.
1) 기존 라이브러리 등 Vue3와 Vue2의 사용 방법이 달라진 것들이 존재하고 학습 비용이 듦
2) 관련 자료 정보의 양의 차이
3) 일부 라이브러리는 Vue3를 지원을 안 함
위 이유로 아직까지는 Vue2를 이용하는 것이 좋을 것이라고 느꼈다.
```

### 어려웠던 점
1) 인원 교체
```
백엔드 3명, 프론트 3명이었던 팀 중 프론트 2명이 나가고 추후 늦게 새로운 프론트 인원 1명이 들어오게 되었다.
프론트에서만 총 3명의 인원이 나가고 들어갔기 때문에 기존 기획하고 설계한 프론트 업무는 나에게 집중됐다.
6주라는 제한된 시간 안에 구현을 해야했기때문에 밤을 새가며 개발에 몰두했다.
또한, 새로운 기술스택인 Vue3와 Stomp 양방향 소켓통신을 익히고 사용하느라 더 힘든 개발이었다.
다행히 비교적 여유로운 인원이 발표 준비, 배포를 맡아주고 다른 백엔드 인원들도 API를 미리미리 만들어줘서 프론트 개발에 집중할 수 있었다.
기간 안에 끝내기 힘들다고 생각한 업무는 팀원들의 배려로 무사히 끝낼 수 있었으며 협업의 중요성을 느꼈다.
```
2) Stomp 양방향 통신
```
실시간 DM과 단체채팅을 구현하기 위해 Stomp 웹 소켓을 사용했다. Stomp는 양방향 통신이 가능한 웹 소켓으로 subscribe/send로 클라 <-> 서버 통신이 가능하다.
때문에 사용자는 쉽게 메세지를 실시간으로 주고 받을수 있다.
대신, 주의해야 할 점은 Subscribe의 코드 구현 시점이다.
유저가 로그인하고 플랫폼의 각 페이지에 접근할 때 각 컴포넌트의 요소가 렌더링되고 코드가 돌아간다.
따라서 subScribe의 위치에 따라 유저는 메세지를 중복해서 받을 수도, 못 받을 수도, 뒤늦게 받을 수 있다. 처음 학습하는 입장에서 이 부분때문에 애를 먹었다.
```
